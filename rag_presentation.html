Retrieval-Augmented Generation

A hands-on guide to building intelligent, knowledge-grounded AI systems — with real implementation code.

What is RAG?

RAG (Retrieval-Augmented Generation) is a technique that combines a large language model (LLM) with a dynamic retrieval mechanism over an external knowledge base. Instead of relying purely on what the model learned during training, RAG fetches relevant context at inference time and feeds it to the model alongside the user's question.

📚
External Knowledge
Documents, PDFs, and text files stored outside the model — always up to date.
🔍
Semantic Search
Vector similarity retrieves the most relevant chunks for any user query.
🤖
Grounded Generation
The LLM generates answers based on retrieved facts — reducing hallucinations.

Why do we need RAG?

Standalone LLMs have a static knowledge cutoff — they cannot access information created after training. They also tend to hallucinate confidently when asked about topics not well-covered in training data. RAG solves both problems.

Plain LLM

  • Knowledge frozen at training time
  • Cannot cite sources
  • Hallucinations on niche topics
  • Cannot access private documents
  • Expensive to retrain for updates

LLM + RAG

  • Knowledge updated via document uploads
  • Answers grounded in retrieved chunks
  • Reduced hallucinations
  • Works on private/proprietary data
  • No retraining required

@data_router.post("/upload/{dir_name}")
async def upload_file(request, dir_name, file, app_settings):
    project_model = ProjectModel(db_client=request.app.mongodb)
    project = await project_model.get_project_or_create_one(project_id=dir_name)

    is_valid, error_msg = DataController().validate_file(file)
    if not is_valid:
        return JSONResponse(status_code=400, content={"error_msg": error_msg})

    file_path = FileController().get_file_path(dir_name) # creates dir if missing
    async with aiofiles.open(file_path, 'wb') as out_file:
        while chunk := await file.read(app_settings.FILE_CHUNK_SIZE):
            await out_file.write(chunk)

# DataController.py — validation
def validate_file(self, file: UploadFile) -> bool:
    if file.content_type not in self.app_settings.FILE_ALLOWED_EXTENSIONS:
        return False, "Invalid file type"
    if file.size > self.app_settings.FILE_MAX_SIZE_MB * 1024 * 1024:
        return False, "File size exceeds limit"
    return True, "success"
Theory → Implementation
The DataController.validate_file() enforces allowed extensions and max file size from config. The FileController.get_file_path() creates the project directory automatically. Files are streamed in chunks (not loaded into memory at once) using aiofiles — this is important for large files.

Text Chunking

Theory
LLMs have limited context windows. Feeding an entire document as context is impractical and dilutes relevance. Chunking splits documents into overlapping segments of a fixed token/character size. Overlap between consecutive chunks prevents information loss at boundaries — a key detail missed in naive implementations.
Document → overlapping chunks
Chunk 1: tokens 0–100
Chunk 2: tokens 80–180 ← 20-token overlap
Chunk 3: tokens 160–260 ← 20-token overlap
from langchain.text_splitter import RecursiveCharacterTextSplitter

def process_files(self, file_content, file_name,
                   chunk_size: int=100, overlap: int=20):
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,   # max chars per chunk
        chunk_overlap=overlap,   # chars shared with next chunk
        length_function=len
    )
    chunks = text_splitter.create_documents(
        [doc.page_content for doc in file_content],
        metadatas=[doc.metadata for doc in file_content]
    )
    return chunks

def get_file_loader(self, file_name: str):
    ext = self.get_file_extension(file_name)
    if ext in ['.txt']:
        return TextLoader(file_path)
    if ext in ['.pdf']:
        return PyMuPDFLoader(file_path)  # preserves layout
    return None
Theory → Implementation
RecursiveCharacterTextSplitter from LangChain tries to split on paragraph breaks first, then sentences, then words — preserving semantic coherence. The chunk_size and overlap are configurable per API call. PyMuPDFLoader is used for PDFs to preserve text order across pages.

Embeddings

Theory
An embedding model converts text into a dense numeric vector (e.g. 768 or 1536 dimensions) that captures semantic meaning. Two semantically similar sentences — even if worded differently — will have vectors that are close in high-dimensional space. This enables "meaning-aware" search, unlike keyword matching.
"The cat sat on the mat"
→
Embedding Model
→
[0.23, -0.51, 0.88 ...]
Vector of ~768–1536 floats representing meaning
def push_data_to_index(self, project, chunks, do_reset=0):
    collection_name = self.create_collection_name(project.id)

    text  = [c.chunk_text     for c in chunks]
    metadata = [c.chunk_metadata for c in chunks]

    # embed each chunk as a "passage" (asymmetric embedding)
    vectors = [
        self.embedding_client.embed(text=t, doc_type="passage")
        for t in text
    ]

    self.vectordb_client.create_collection(
        collection_name,
        self.embedding_client.embedding_size  # dimension of vectors
    )
    self.vectordb_client.add_documents(
        collection_name=collection_name,
        texts=text,
        vectors=vectors,
        metadata=metadata
    )
Theory → Implementation
Note the doc_type="passage" flag — many modern embedding models (like E5, BGE) use asymmetric embedding: passages and queries are embedded differently to improve retrieval quality. The vector dimension is stored as embedding_size and used when creating the vector DB collection.

Vector Database

Theory
A vector database (e.g. Qdrant, Pinecone, Chroma) is purpose-built for storing and querying high-dimensional vectors efficiently. Unlike a relational DB, it supports Approximate Nearest Neighbor (ANN) search — finding the top-K vectors most similar to a query vector in milliseconds, even at millions of entries.

[0.2, 0.9, ...]

[0.7, 0.1, ...]

[0.3, 0.8, ...]
Chunk D
[0.5, 0.4, ...]
Query
[0.3, 0.8, ...]
Chunk E
[0.1, 0.3, ...]
Query → nearest neighbor = Chunk C (cosine similarity)
def create_collection_name(self, project_id):
    return f"collection_{project_id}".strip()

def reset_vector_db_collection(self, project: Project):
    collection_name = self.create_collection_name(project.id)
    return self.vectordb_client.delete_collection(collection_name)

def get_vector_db_collection_info(self, project: Project):
    collection_name = self.create_collection_name(project.id)

Theory → Implementation
Each project gets its own isolated vector collection, named from the MongoDB project _id. This is multi-tenancy by design — different users' documents don't interfere. The do_reset flag triggers a full collection delete before re-indexing, allowing clean re-ingestion.

Similarity Search

Theory
At query time, the user's question is embedded into a vector using the same model (but with doc_type="query"). The vector DB then finds the top-K most similar stored vectors using cosine similarity or dot product. These corresponding chunks become the context for the LLM.
User: "What is X?"

→

Return chunk texts
def search_by_vector(self, project: Project,
                      text: str, top_k: int=5) -> list:
    collection_name = self.create_collection_name(project.id)

    # Note: doc_type="query" — asymmetric vs "passage" at index time
    query_vector = self.embedding_client.embed(
        text=text, doc_type="query"
    )

    results = self.vectordb_client.search_by_vector(
        collection_name=collection_name,
        query_vector=query_vector,
        top_k=top_k
    )

    if not results:
        return False
    return results  # list of scored chunk objects

# nlp.py — exposed via API with configurable top_k
@nlp_router.post("index/search/{project_id}")
async def search_by_vector(request, project_id, search_request):
    results = nlp_controller.search_by_vector(
        project=project,
        text=search_request.text,
        top_k=search_request.top_k
    )
Theory → Implementation
The top_k parameter is user-controlled per request — more results provide broader context but increase prompt length. The asymmetric embedding (doc_type) is crucial: query vectors and passage vectors live in slightly different representation spaces optimized for retrieval accuracy.

Prompt Augmentation

Theory
Prompt augmentation (also called context stuffing or grounding) is the process of constructing a structured prompt that includes: (1) a system instruction defining the assistant's role, (2) the retrieved document chunks as numbered references, (3) the user's original question, and (4) a footer instruction telling the model how to use the context.
[SYSTEM] You are a helpful assistant...
[DOC 1] Retrieved chunk text...
[DOC 2] Retrieved chunk text...
[USER QUERY] What is X?
[FOOTER] Answer using only the above documents.
def answer_rag_question(self, project, query, top_k=5):
    relevant_docs = self.search_by_vector(project, query, top_k)

    # Load prompt templates from template_parser
    system_prompt   = self.template_parser.get("rag", "system_prompt")
    footer_prompt   = self.template_parser.get("rag", "footer_template_prompt")

    # Build numbered document section from retrieved chunks
    documents_prompt = "\n".join([
        self.template_parser.get("rag", "document_prompt", {
            "doc_id": idx+1,
            "text": doc.text,
        })
        for idx, doc in enumerate(relevant_docs)
    ])

    # Combine all parts into final prompt
    full_prompt = "\n\n".join([
        documents_prompt,
        f"## User Query:\n{query}",
        footer_prompt
    ])

    chat_history = [self.generation_client.construct_prompt(
        query=system_prompt, role="system"
    )]
    answer = self.generation_client.generate_response(
        prompt=full_prompt, chat_history=chat_history
    )
    return answer, full_prompt, chat_history
Theory → Implementation
The system uses a template engine (template_parser) to separate prompt text from code — a best practice for prompt management. The chat history starts with the system prompt as a "system" role message. The full prompt and history are returned to the API caller for transparency and debugging.

LLM Generation

Theory
The final step passes the augmented prompt to an LLM (GPT-4, Claude, Mistral, etc.) for answer generation. The key difference from vanilla LLM usage: the model is explicitly instructed to answer from the provided documents only. This "grounding" constraint dramatically reduces hallucinations and keeps answers factually anchored to your data.

@nlp_router.post("/index/answer/{project_id}")
async def answer_rag(request, project_id, search_request):
    project = await project_model.get_project_or_create_one(project_id)

    nlp_controller = NlpController(
        vectordb_client=request.app.vector_db_provider,
        generation_client=request.app.generation_provider,
        embedding_client=request.app.embedding_provider,
        template_parser=request.app.template_parser
    )

    answer, full_prompt, chat_history = nlp_controller.answer_rag_question(
        project=project,
        query=search_request.text,
        top_k=search_request.top_k
    )

    if not answer:
        return JSONResponse(status_code=400, content={"results": "Failed"})

    return JSONResponse(status_code=200, content={
        "answer": answer,
        "full_prompt": full_prompt,     # for debugging/transparency
        "chat_history": chat_history    # full conversation context
    })
Theory → Implementation
All four provider clients (vectordb_client, generation_client, embedding_client, template_parser) are injected from request.app — set up at startup via FastAPI's lifespan context. This dependency injection pattern makes each provider swappable (e.g. swap OpenAI for a local model) without changing controller code.

Common pitfalls & best practices

⚠
Not resetting the collection before re-indexingIf do_reset=0 and you re-index the same document, you get duplicate chunks. Always set do_reset=1 when re-processing a project.
⚠
Using the same embed type for queries and passagesAsymmetric models like E5 require doc_type="query" at search time. Using "passage" for queries degrades recall.
⚠
Fetching all chunks at onceThe code correctly paginates with page_size=50 in the push endpoint. Loading millions of chunks at once causes memory exhaustion.
✓
Bulk-write for chunk insertionThe ChunkModel.insert_many_chunks() method uses MongoDB's bulk_write with batching — significantly faster than inserting one-by-one.
✓
Provider injection via FastAPI app stateAll external clients (LLM, embedder, vector DB) are attached to request.app at startup, making them swappable and testable.

Exercise 1 — Chunking

The current ProcessController only supports .txt and .pdf files. How would you extend it to support .docx files? Which class would you modify?

Add a new condition in get_file_loader() checking for .docx extension. Use LangChain's Docx2txtLoader or UnstructuredWordDocumentLoader. No other class needs changing — the file loader pattern isolates format-specific logic.
Exercise 2 — Retrieval quality

A user reports that RAG answers are often missing relevant information. Given that top_k=5 is the default, what are two things you could change in the codebase to improve recall?

1. Increase top_k (more chunks = more context, but longer prompt). 2. Reduce chunk_size so chunks are more focused on single ideas, improving relevance. 3. Experiment with a better embedding model with higher dimensionality.
Exercise 3 — Architecture

The NlpController receives clients via constructor injection. What testing benefit does this give you, and how would you write a unit test for search_by_vector()?

Dependency injection allows mocking: create mock objects for embedding_client and vectordb_client that return predictable values. Pass them to NlpController() in your test. This lets you test the controller logic in isolation without a real vector database or embedding API.
